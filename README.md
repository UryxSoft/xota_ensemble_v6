## Overview

This system detects whether a piece of text was written by a human or generated by one of six major AI models. It uses a **three-model soft-voting ensemble** trained with a custom **Multi-Level Contrastive Loss (MCL)** that explicitly enforces:

- A strong Human ↔ AI decision boundary (δ=3.0 weight)
- Per-model AI clustering (GPT samples cluster together, Claude samples cluster together, etc.)
- Inter-model AI separation (GPT embeddings pushed away from Claude embeddings)

Every text is prefixed with a **Flesch-Kincaid Grade Level** score before being fed to the models, giving them an explicit readability signal. AI models tend to write at a narrower, more uniform reading level than humans.

## Architecture

```
Input text
    │
    ├─ [FK_SCORE: 8.3] text...        ← Flesch-Kincaid prefix
    │
    ├──────────────────────────────────────────────────────
    │         3-Model Ensemble (sequential training)
    │
    ├─ DeBERTa-v3-base ──────────────── softmax probs (7)
    │    Disentangled attention
    │    lr=1e-5 · bs=48 · effective_batch=96
    │
    ├─ ELECTRA-base ─────────────────── softmax probs (7)
    │    Replaced-token detection pretraining
    │    lr=2e-5 · bs=64 · effective_batch=128
    │
    └─ ModernBERT-base ──────────────── softmax probs (7)
         SDPA · Flash Attention 2
         lr=2e-5 · bs=64 · effective_batch=128
    │
    └─ Soft Voting (equal weights 33/33/33)
         P_ensemble(c) = mean([P_deberta, P_electra, P_modernbert])
         ŷ = argmax P_ensemble
```

### Loss Function

Each training step combines two losses:

```
Loss = MCL(CLS_embedding, labels) + CrossEntropy(logits, labels, class_weights)
```

**MCL** operates on the CLS token embedding of the last hidden layer:

| Level | Positives | Negatives | Weight |
|---|---|---|---|
| L1 — Human/AI boundary | Human ↔ Human | Human ↔ AI | δ = 3.0 |
| L2 — Same model cluster | GPT ↔ GPT, Claude ↔ Claude | AI ↔ different AI | α = 1.0 |
| L4 — Inter-model separation | Different AI models | AI ↔ Human | γ = 1.0 |

Temperature τ = 0.3 (increased from 0.1 in v5.3 for smoother gradients).

1. If `deberta_final/config.json` exists → skip DeBERTa entirely
2. If `deberta_checkpoints/checkpoint-XXXXX/` exists → resume from last checkpoint
3. Otherwise → train from scratch

Key metrics to watch:

| Metric | Expected behaviour |
|---|---|

| `train/learning_rate` | Ramp up during warmup (~8k steps), then linear decay |
| `train/grad_norm` | 100–300 range normal for MCL. Spikes >500 are acceptable if isolated |

**Loss convergence pattern (DeBERTa, 4M rows):**

```
Step  1,000 → ~19.6   (warmup phase)
Step  5,000 → ~15.8   (post-warmup acceleration)
Step 10,000 → ~12.7   (end of warmup, plateau begins)
Step 40,000 → ~11.9   (epoch 1 plateau)
Step 83,672 → ~11.8   (final)
```

The MCL loss starts high (~19–20) because the contrastive term with τ=0.3 dominates early training. This is expected behaviour.

### Single text

```python
from detector_v6 import EnsembleDetector

detector = EnsembleDetector(
    ensemble_dir="/content/drive/MyDrive/your_project/ensemble_v6/",
    use_statistical_features=True,   # uses StylometricProfiler if available
    confidence_threshold=0.60,
    device=None,                      # auto-selects best GPU or CPU

result = detector.detect("Your text here...")

print(result.prediction)          # "Human" | "AI" | "Unknown"

print(result.model_family)        # ModelFamily.FRONTIER_OPENAI | etc.

print(result.statistical_features)# dict: burstiness, vocab_richness, etc.
```

## Training Details

### Optimizer

`adamw_torch_fused` — the PyTorch fused implementation, ~15% faster than standard AdamW on CUDA. Parameters: `weight_decay=0.01`.

### Precision

`bf16=True` + `tf32=True` on all models. BFloat16 is numerically more stable than float16 for fine-tuning with contrastive losses.

### Dataloader

`num_workers=2` + `pin_memory=True` + `prefetch_factor=2` — overlaps data loading with GPU compute to prevent the GPU from idling between batches.

### Checkpointing strategy

`save_strategy="steps"` with `save_steps = max(10000, total_steps // 12)` — approximately 12 checkpoints per model training run. `save_total_limit=3` keeps only the 3 most recent on local SSD. All checkpoints are also copied to Drive by `StabilityCallback.on_save()`.

### No eval during training

`eval_strategy="no"` (nuclear mode) — eliminates ~15% of compute that would be spent on mid-training evaluation passes. Full evaluation runs after all models are trained in Phase 4.

## Troubleshooting

### "Los siguientes datasets tokenizados NO existen"

The training script found no tokenized data in Drive. Run preprocessing first:

```bash
# CPU runtime
!python ensemble_v6_preprocess.py
```

### Loss stuck at ~19–20 in early steps

Expected. The MCL contrastive term dominates early training and starts high with τ=0.3. The loss should begin dropping sharply after the warmup phase (~step 8,000 for DeBERTa). If still at 19+ past step 15,000, check that `CLASS_WEIGHTS` matches your actual dataset distribution.

### grad_norm spike > 500

Isolated spikes (1–2 occurrences) during training are normal for MCL with diverse batches. If grad_norm stays consistently above 400 for more than 5,000 steps, add gradient clipping:

```python
# In TrainingArguments:
max_grad_norm=1.0,
```

### torch_compile error on DeBERTa

DeBERTa's disentangled attention uses dynamic shapes that can cause compile failures. It is intentionally set to `torch_compile=False` in the config. If you see compile errors on ELECTRA or ModernBERT, set their `torch_compile` to `False` as well.

### W&B "Finishing previous run"

Harmless. W&B closes the run from the previous interrupted session before starting a new one. Training continues normally.

### Colab session killed before training completes

Re-run `ensemble_v6_train.py`. The script:
1. Skips models with a completed `config.json` in Drive
2. Resumes the in-progress model from its latest Drive checkpoint
3. Continues with remaining models

No data is lost.

### VRAM shows only 1–2 GB in logs

The `StabilityCallback` logs VRAM between steps, not during the forward pass. The actual peak VRAM (~22 GB) occurs inside the step and is not captured by the log. If you want to see the true peak, add `torch.cuda.max_memory_allocated()` to the callback.
